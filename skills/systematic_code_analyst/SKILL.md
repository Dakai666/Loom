# 系統化程式碼分析技能（Systematic Code Analyst）

透過結構化掃描，系統性地理解陌生或複雜的程式碼庫，輸出有洞見的分析報告。

---

## 📁 技能結構

```
skills/systematic_code_analyst/
└── SKILL.md           ← 本技能定義
```

---

## 🎯 核心原則（四不）

1. **不先入為主** — 不要假設自己懂這個專案在幹嘛，先證明再說
2. **不追細節** — 先畫地圖，再定點挖掘；拒絕一頭栽進函數裡
3. **不混事實與推測** — 明確區分「我確定的事實」和「我的假設」，不偽裝確定性
4. **不只描述表面** — 說出「這個設計解決了什麼問題」而非「這個檔案包含什麼函數」

---

## 🔄 通用工作流程（四層掃描）

### 第一層：結構探測（Structural Survey）
**目標：建立整體地圖，不要深入細節**

用 `list_dir` 觀察：
- 頂層目錄結構（目錄命名通常揭示模組組織方式）
- 主要原始碼位置（`src/`、`lib/`、`core/`、`cmd/`？）
- 設定檔種類與數量（有多複雜？常見於：`pyproject.toml`、`package.json`、`Cargo.toml`）
- 入口點位置（CLI 主體？API server？library？）

觀察細節：
- 命名 convention 一致性（全小寫 kebab？駝峰？層次分明？）
- 目錄深度（淺=可能簡單好維護，也可能功能少；深=可能職責分離良好，也可能過度工程）
- 測試位置（`tests/` 在同層？`__tests__/`？還是找不到？）

### 第二層：依賴解析（Dependency Resolution）
**目標：識別核心資料結構與模組之間的依賴方向**

用 `read_file` 重點掃描：
- 主要的 class / struct / type 定義（出現在最多檔案的通常是核心）
- `import` / `use` / `require` 語句（第三方 library 使用方式）
- 枚舉全域搜尋（`enum`/`const`/`DATABASE`/`CONFIG`）— 通常揭示系統邊界
- 主要函數簽名（`def ` / `fn ` / `func ` 行）— 不要讀 body，看簽名就好

畫出心智模型：
- 誰是核心（被最多其他模組 import 的）？
- 誰是邊緣（幾乎不依賴其他模組，或只被少數使用）？
- 有沒有循環依賴？（A→B→C→A）
- 第三方 library 的選擇透露什麼？（例：用 requests 不用 httpx → 可能老牌 Python 專案）

### 第三層：行為模式（Behavioral Pattern）
**目標：識別設計決策背後的邏輯**

觀察：
- 錯誤處理策略：
  - 全域 catch-all？（`except Exception` 藏在哪層）
  - 每層自負責任？
  - 統一的 error type 還是到處字串 error？
- I/O 封裝層次：
  - database access 有統一路由？還是到處散落 raw query？
  - 檔案操作有統一抽象？
- 狀態管理：
  - 全域變數多不多？
  - 狀態集中在一個地方還是分散各處？
- 配置外部化：
  - 設定是 hardcode 還是從環境讀取？
  - 有多少個不同的設定來源？

### 第四層：評估與假說（Evaluation & Hypothesis）
**目標：說出人話，讓讀者不用再問「所以呢？」**

在完成前三層之後，用以下框架回答：

#### 這個設計解決的問題（確定的）
不要描述「做什麼」，要描述「解決什麼」：
- ❌ 「這個系統用 FastAPI 做 API」
- ✅ 「這個系統需要一個高並發 HTTP API，所以選了非同步框架」

#### 這個設計在什麼條件下會失敗（推測，標記清楚）
- 「若流量成長 10 倍，目前的資料庫連線池設定會是第一個瓶頸」
- 「若要支援多租戶，目前的 schema 設計需要大改」（基於「我看到的事實」的合理推測）

#### 技術債訊號
看起來不合理但可能是時間壓力下的取捨：
- 「這段邏輯重複了三次，可能是赶 deadline 没来得及重構」
- 「變數命名不精確，可能是初期 rapid prototyping 的痕跡」

#### 重構機會（最小改動最大收益）
- 「把 `utils.py` 拆出兩個職責明確的 module，代價低但長期可讀性提升顯著」
- 「錯誤處理統一到一個 module，現在散落各處修改成本高」

## 📤 輸出格式

**每次分析都使用這個固定格式**，確保結構一致、容易比較。

```markdown
# 🔍 [專案名稱] — 系統化分析報告

## 📂 結構快照
（2-3 句話，說清楚這是什麼、在做什麼、整體輪廓）

## 🔀 依賴拓撲
（核心模組是什麼、依賴方向、主要第三方 library 的角色）
（可用簡短文字描述，不必畫圖）

## ✅ 觀察到的優點
- **具體指出**，不含糊，附理由
- 說「為什麼這是好設計」，不只是「好」

## ⚠️ 觀察到的疑慮
- **說明理由**，不是情緒批評
- 標記確定性：「確定」vs「推測」
- 若有技術債跡象，說出可能的原因

## 💡 學習點
- 如果你要接手這個專案，會先做什麼？
- 這個設計有什麼值得帶走用在其他專案的？

## 🎯 適合場景
- 這個專案適合用在什麼條件下？
- 什麼情況下**不應該**用它？

---

*本報告由 systematic_code_analyst skill 生成*
```

## ⚙️ 使用範例

### 自我健檢（分析 Loom 自身）
當使用者丢一個本地路徑或直接要求分析目前專案時：
1. 先 `list_dir` 看目錄結構
2. 根據副檔名鎖定主要語言（`.py` → Python）
3. 讀關鍵檔案：`SOUL.md`、`loom.toml`、`loom/__init__.py`、`loom/core/` 下的主要 module
4. 用四層掃描框架產出報告

### 開源專案分析
當使用者提供 GitHub URL 或 clone 後的本體時：
1. 若有 URL，先用 `fetch_url` 取得 repo 基本資訊或 README
2. `list_dir` 觀察目錄結構
3. `read_file` 讀 README.md、主要的 entry 檔案
4. 用四層掃描框架產出報告

## 🛠️ 工具使用策略

| 工具 | 使用時機 |
|------|---------|
| `list_dir` | 結構探測階段，所有目錄 |
| `read_file` | 依賴解析與行為模式階段，重點檔案 |
| `run_bash` | 找特定副檔名檔案列表：`find . -name "*.py" \| head -20` |
| `web_search` | 確認某個 library 的定位或社群活躍度 |
| `fetch_url` | 取得 GitHub README 或官方文件 |

**嚴格控制 read_file 數量**：不要讀超過 15 個檔案，否則會陷入細節泥沼。
優先讀：目錄結構描述的入口點 + README + 最大的幾個 module 的開頭。

---

## ⚠️ 紀律提醒

- 不要試圖讀懂每一行 code，重點是理解整體設計邏輯
- 不要幫忙重構，除非使用者明確要求，那是另一個技能
- 分析結論要敢說「我確定的」vs「我推測的」，不確定就說不知道
- 如果資訊不足（封閉原始碼、缺少文件），誠實說「無法完整評估」
