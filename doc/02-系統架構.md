# 系統架構

Loom 的架構分為七層，每層有明確的職責邊界與標準化接口。低層為上層提供服務，上層依賴但不應直接穿透至隔離的鄰層。

---

## 七層總覽

```
┌─────────────────────────────────────────────────────────────┐
│  Platform Layer         對外介面：CLI / TUI / REST / Bot    │
├─────────────────────────────────────────────────────────────┤
│  Cognition Layer        思考樞紐：LLM 路由 / Token 配額     │
├─────────────────────────────────────────────────────────────┤
│  Harness Layer  ←───  骨幹：工具生命週期 / Middleware       │
├─────────────────────────────────────────────────────────────┤
│  Memory Layer           基底：四種記憶 / 搜尋 / 索引       │
├─────────────────────────────────────────────────────────────┤
│  Task Engine            並行：DAG 圖 / 排程器               │
├─────────────────────────────────────────────────────────────┤
│  Autonomy Engine        自主：觸發器 / 決策器 / 通知       │
├─────────────────────────────────────────────────────────────┤
│  Extensibility Layer    生態：Lens / Plugin / 技能匯入     │
└─────────────────────────────────────────────────────────────┘
```

---

## Platform Layer（最上層）

**職責**：提供 Loom 與外部世界的交互介面。

| 介面 | 說明 |
|------|------|
| **CLI** | `loom chat` / `loom memory` / `loom reflect` / `loom autonomy` |
| **TUI** | Textual 終端圖形介面，雙欄佈局，streaming 輸出 |
| **REST API** | FastAPI 伺服器，HTTP endpoints 操作 memory / 接收 webhook 回覆 |
| **Bot** | Telegram Bot / Discord Bot，接收指令與回覆 |

**不應做**：直接操作 LLM 或 Memory 業務邏輯。

---

## Cognition Layer（思考層）

**職責**：管理 LLM 的選擇、調用、與上下文控制。

### 三個核心子系統

| 子系統 | 檔案 | 說明 |
|--------|------|------|
| **LLM Router** | `cognition/router.py` | 根據模型前綴路由到正確的 Provider（MiniMax / Anthropic）|
| **Context Budget** | `cognition/context.py` | 追蹤 token 使用量，達閾值自動 `_smart_compact()` |
| **Reflection API** | `cognition/reflection.py` | 自我觀察：session summary、tool success rate、skill health |

### 訊息格式約定

```
Internal Format（Loom 內部）
    ↓ Provider._to_anthropic_messages()
Anthropic Wire Format（發送給 Anthropic API）
    ↓
Anthropic Response
    ↓
Internal Format（回到 Loom 內部流程）
```

**為什麼這麼設計？**
Loom 內部統一使用 OpenAI-canonical 格式（`role` / `content` / `tool_calls`），Anthropic 格式轉換封裝在 Provider 內部。這樣 Router 可以無感知地切換 Provider。

**不應做**：在 Router 之外出現 provider 特定的 message 格式假設。

---

## Harness Layer（骨幹層）⭐

**職責**：工具生命週期的完整管理。

### Middleware Pipeline

每一次 tool call 都會依序通過三個 Middleware：

```
Tool Call 發起
    ↓
LogMiddleware        →  Rich 格式化輸出到終端
    ↓
TraceMiddleware      →  計時 + 寫入 Episodic Memory
    ↓
BlastRadiusMiddleware →  依 Trust Level 決定是否暫停等確認
    ↓
實際工具執行
    ↓
TraceMiddleware（回傳）→  tool result 寫入 episodic
    ↓
BlastRadiusMiddleware（回傳）→  釋放 locks / 更新 session 狀態
```

### ToolRegistry

- 所有工具（內建 + 外掛）在啟動時註冊到 ToolRegistry
- 提供 `to_anthropic_schema()` / `to_openai_schema()` 雙格式輸出
- 工具定義包含：名稱、描述、參數 schema、Trust Level

**Harness 與其他層的關係**：
- Memory Layer 的 EpisodicMemory 在 TraceMiddleware 中被寫入
- Cognition Layer 的 ReflectionAPI 消費 TraceMiddleware 收集的數據
- Autonomy Engine 的 ActionPlanner 透過 Harness 執行決策後的工具

**不應做**：在 Harness 之外直接呼叫工具函數繞過 Middleware。

---

## Memory Layer（基底層）

**職責**：四種記憶的持久化、檢索、與維護。

### 四種記憶類型

| 類型 | 用途 | 生命週期 |
|------|------|----------|
| **Episodic** | 順序事件紀錄 | Session 內，結束後壓縮 |
| **Semantic** | 結構化事實 | 長期 |
| **Procedural** | 技能（Skill Genome）| 長期，有版本 |
| **Relational** | 關係三元組 | 長期 |

### 後端

所有記憶使用 **SQLite WAL 模式**：
- 同一进程可并发读取
- 写入不阻塞读取
- 適合本地長時間運行的 daemon

### 搜尋能力

| 階段 | 機制 |
|------|------|
| Phase 4（已實作）| BM25 / TF-IDF 關鍵字加權 |
| Phase 5（已實作）| MiniMax Embedding cosine similarity（第一優先）|

**不應做**：在 Session 內累加 token 追蹤（應用 replace 語義）。

---

## Task Engine（並行層）

**職責**：將多步任務分解為 DAG 並行執行。

```
任務描述
    ↓ LLM 規劃
DAG（節點=子任務，邊=依賴）
    ↓ Kahn's Topological Sort
分層（levels）+ 可並行組（parallel_groups）
    ↓ asyncio.gather 同層並行
結果收集
```

### 與 LoomSession 的整合

- `_dispatch_parallel()` 接收 TaskGraph，輸出可並行的工具列表
- `_all_authorized()` 檢查所有工具是否已預授權
  - 是 → asyncio.gather 並行執行
  - 否 → 循序執行（避免 CLI 平行確認混亂）

**不應做**：將 stop_on_failure 的下游節點標記為 SKIP 後繼續執行。

---

## Autonomy Engine（自主層）

**職責**：在無人類觸發的情況下自主決策並行動。

### 三種觸發器

| 觸發器 | 觸發條件 |
|--------|----------|
| **CronTrigger** | 符合 cron 表達式的時間 |
| **EventTrigger** | 系統內部 emit 事件 |
| **ConditionTrigger** | Python lambda 條件為真 |

### 決策管道

```
Trigger fires
    ↓
Context Assembly（Memory Index + fire_context）
    ↓
Action Planner
    ├─ safe     → EXECUTE
    ├─ guarded  → NOTIFY 或直接 EXECUTE
    └─ critical → HOLD（每次確認）
    ↓
LoomSession 執行
    ↓
NotificationRouter 通知結果
```

### Daemon 模式

`loom autonomy start` 啟動背景 loop，每 60 秒評估一次觸發器狀態。

**不應做**：在非 daemon 模式下自行發起定時檢查。

---

## Extensibility Layer（生態層）

**職責**：讓 Loom 能吸收外部框架的設計精華。

### 三大擴充機制

| 機制 | 說明 |
|------|------|
| **Lens** | 將外部工具/技能/提示格式轉換為 Loom 標準（HermesLens / OpenAIToolsLens）|
| **Plugin** | 可同時攜帶 tools / middleware / lenses / notifiers 的完整擴充單元 |
| **Skill Import** | 將外部 skills 匯入 Procedural Memory，帶 schema 審查 + confidence gate |

### LensRegistry

啟動時自動掃描 `loom/extensibility/` 目錄下的所有 Lens 子類，統一路由。

**不應做**：讓 Plugin 直接修改 Core 類的內部狀態——透過 `install_into(session)` 標準接口注入。

---

## 層間依賴圖

```
Platform ──依賴──▶ Cognition
                       │
                       ▼
                  Harness ◀──被使用── Task Engine
                       │
                       ▼
Memory ◀──支撐── Autonomy
                       │
                       ▼
              Extensibility
```

**重要原則**：
- Memory Layer 是幾乎所有其他層的底層支援，但不依賴任何上層
- Harness Layer 是工具呼叫的必經之路，Task Engine 和 Autonomy 都透過它執行
- Cognition Layer 位於 Platform 與其他層之間，負責所有 LLM 通訊

---

## 設計約束

| 約束 | 理由 |
|------|------|
| Harness 必須是所有工具呼叫的守門人 | 確保 Middleware / Trust / Audit 的一致性 |
| Memory 使用 SQLite WAL 而非網路資料庫 | 適合本地 daemon 的延遲特性 |
| 內部訊息格式統一為 OpenAI-canonical | Provider 可插拔，Router 無感知 |
| SOUL.md 限制在 ~300 tokens | 永遠駐留 context，不能讓它膨脹 |
| Plugin 不直接修改 Core 內部狀態 | 透過標準接口維護可組合性 |
