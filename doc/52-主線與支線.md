## 1. 共識主線（劃星採納）

以下方向被 Loom 確認為未來 6-12 個月主線，每條都對應 milestone 與 issue。

### 1.1 中樞神經 vs 免疫系統 — 最高層 mental model

doc/50 提出 Loom 處於「器官齊全等神經系統與免疫系統」階段。本文件採納為最高層敘事框架：

- **中樞神經 = AgentLedger** — 統一事件時間線（見 §1.4）
- **免疫系統 = Sandbox + Capability Firewall + Memory Governance 演化治理**（見 §1.2、§1.3）

所有主線都應該回扣到「補齊神經」或「補齊免疫」這兩個方向之一。

### 1.2 Quest A — Sandbox Wall

把 Loom 從 policy/tripwire 級安全推進到 isolation-backed execution。

**核心動作**：
- `ExecutionBackend` 抽象（先抽象再加硬，預設 local subprocess，允許 docker / sandbox-exec / firejail backend）
- workspace mount allowlist、network egress policy、env secret projection
- per-tool capability declaration（fs/network/env/process）

**對應 issues**：#29（cornerstone）、#214（whitelist mode）

### 1.3 Quest D — Skill Evolution Arena（演化治理）

doc/50 點中的「Memory Governance 從寫入治理 → 演化治理」是 Loom 最該深化的護城河。skill mutation / dreaming / reflection 必須能證明「變好了」。

**核心動作**：
- replay corpus：失敗 trace、成功 trace、edge cases
- skill candidate promotion 必須伴隨 metric 改善
- dreaming triples 有 usefulness / contradiction / staleness 評估
- judge verdict 有 false positive / false negative corpus

**對應 issues**：#296（技能健康儀表板 epic）、#295（co-occurrence）、#292（usage_count）

### 1.4 Quest B — Unified Event Ledger

`ActionRecord` / `ExecutionEnvelope` / `SessionLog` / `TaskList` / `MemoryWrite` / `JudgeVerdict` 都在描述同一件事。把它們收斂成 `AgentLedger`，所有平台 UI 與 memory compression 從 ledger 投影。

**Loom 自家補強**（doc/50 沒看到的）：
- 不要重新發明 ledger 結構，**把 `ExecutionEnvelope` 升格為 ledger projection 的 first-class view**
- ledger 設計時就要為「Turn Graph Branching」（§3.3）預留 branch-id 與 parent-turn-ref 欄位

**對應 issues**：尚無，先佔 milestone 位置

### 1.5 Memory System v2

三軸 schema + MemoryPulse 鉤子。Phase 0 ontology 已鎖、Hook G + Hook A 已 ship（v0.3.6.0）。

**對應 issues**：#280（主動式記憶鉤子）、#281（三軸分類）

### 1.6 Quest E — Doc Integrity

不做 doc/50 提的「整份 doc AST 掃描」（過度），只 CI **聲稱「已實作」的 symbol path** + version 同步。

**對應 issues**：#313（doc integrity CI 三層方案）

## 2. 共識緩議（過度擔憂或踩到設計線）

以下是 doc/50, 51 的提案，Loom 不採納或降級為觀察項目。**每條都記錄理由，避免日後重新爭論**。

### 2.1 Quest C「Split the Runtime Core」→ 降級為 **Quest C-Watch: Section Hygiene**

**緣由**：doc/50 Risk #1 與 doc/51 Quest C 都建議拆 `LoomSession` 為 4 個 runtime 類。但 Loom 是純 vibe coding 工作流，代碼主要讀者是 agent。檔案行數本身不是壞味道，責任糾纏才是。詳見 `feedback_vibe_coding_code_style` 與 issue #145 的關閉留言。

**改做**：
- session.py 已加 13 個 `# SECTION N` banner（commit `db34a2d`，2026-05-06）
- Quest C-Watch 監控指標：每個 section 內 `self.X` set 的跨 section 交集大小。**指標惡化才考慮拆檔**
- 觀察 banner 是否被新加的方法繞過

### 2.2 Doc/50「Tool Contract Tests Generated From Registry」— 緩議

每個 tool 已有 schema validation；自動生成 contract test 容易產出「表面綠」測試，稀釋真實覆蓋。等出現具體 bug pattern 再選擇性實作。

### 2.3 Doc/50「Operator Replay」— 緩議

想法漂亮但低估了 non-deterministic LLM 的 replay 難度（temperature、cache、上下文殘留）。要做，先定義「同一次 replay」是什麼意思。等 Quest B Ledger 完成後再評估。

### 2.4 Doc/50「Agent Society / Consensus Gate / Debate Protocol」— 緩議

在 Loom 規模下是 over-engineering 風險高地。multi-agent debate 在實證上 ROI 模糊。subagent 維持 bounded、ephemeral、parent-controlled 即可。**等真正出現需要 critic 的高風險決策場景再啟動**。

### 2.5 Doc/51「0-100 精度數值」— 改為定性等第

Power=84 vs 88 的差別其實是 vibe，不是測量。建議 capability sheet 砍掉 0-100 數字，保留 A/B+/B 定性等第與 Evidence 欄。否則下次 review 會掉進「為什麼從 84 變 86」的偽精度爭論。

### 2.6 Doc/51「每月或每個 minor release 重新評估」— 改為事件觸發

cadence 太頻繁會儀式化。建議改為「每 milestone」+「臨時觸發（任何 P0 quest 完成或新 debuff 出現）」。

## 3. Loom 自家創意延伸（doc 50/51 沒看到）

這四條是顧問視角看不到、但跟 Loom 既有基礎盤接得乾淨的真正槓桿點。

### 3.1 Capability Sheet 自動投影（從 GitNexus 生成）

doc/51 的 Capability Sheet 不應該手填。Action Lifecycle 的 Evidence 欄可以從：

- GitNexus query → process count、affected modules
- pytest coverage → 該 capability 對應檔案的測試覆蓋
- `gitnexus_detect_changes` 30 天活躍度
- session log → 該 capability 真實使用頻率

直接生成。**doc/51 自己解決 doc/50 抱怨的 drift 問題 → 一份活的能力儀表板**。

可能的實作：`loom capability sheet --refresh` CLI 子命令。

### 3.2 Buff/Debuff 變成 runtime trade-off 旋鈕

doc/51 的 buff/debuff 目前是描述性。可以變成可開關、有量化 trade-off 的 runtime config：

- 開啟 sandbox = +30 safety / -10ms latency
- 開啟 dreaming = +15 long-term coherence / -8 short-term predictability
- 啟用 verdict judge = +12 hallucination catch / -1 turn overhead

讓 user 看到實際 trade-off 數字而非只是 capability 描述。Quest D 的 eval corpus 同時提供量化基礎。

### 3.3 Belief Git 升級為 Turn Graph Branching

doc/50 提的 Belief Git 主要在 memory 層。可以更激進：**整條 turn 序列可以 branch**。

- 「如果上一輪我選 tool X 而不是 Y」可以從 ledger 反事實重跑
- 把 doc/50 的 Operator Replay（§2.3 緩議）跟 doc/51 Quest D Evolution Arena 接成同一個 substrate
- 與 §1.4 Quest B Ledger 設計時就要為這個保留 branch-id 與 parent-turn-ref

### 3.4 Quest C-Watch — Section Hygiene Lint（取代「拆 god object」舊 rule）

每個 LoomSession section 內部的 `self.X` set 之間交集應該保持小。寫個 lint 跑這個指標：

- 指標惡化 → trigger 警告
- 指標連續 N 個 commit 惡化 → 建議局部抽出
- **指標好 → 即使檔案 5000 行也不拆**

把「god object 必拆」這個經典 rule 換成可量化的 trigger，比硬性原則更 vibe-coding-friendly。
