# Loom 未來改善路線圖

> Codex + GPT-5.5 架構審視草稿  
> 日期：2026-05-06 Asia/Taipei  
> 範圍：`README.md`、`doc/`、`loom/`、`tests/` 的抽樣深讀與結構掃描

---

## 一句話判斷

Loom 已經不是「一個會呼叫工具的 agent」，而是一個正在長出自我治理能力的 agent 作業系統雛形。

它最有價值的方向不是追逐某個模型 API 的新功能，而是把 agent 的行動、記憶、授權、反思、並行與自治都變成可觀測、可審計、可演化的底盤。這個定位非常好，而且比多數 demo-first agent 框架更耐未來。

真正的下一階段挑戰是：Loom 已經有很多正確的器官，現在需要長出「中樞神經系統」與「免疫系統」。前者讓所有事件、記憶、任務與 UI 呈現共享同一條語義時間線；後者讓惡意輸入、錯誤工具、記憶污染、文件漂移與模型幻覺都能被隔離、回溯、矯正。

---

## 觀察依據

本次審視主要看到以下現況：

- `loom/` 內約 127 個 Python 原始檔，`tests/` 約 56 個測試檔，專案已跨過 prototype 階段。
- README 與文件明確定位為 harness-first、memory-native、self-directing。
- 核心模組已形成七層結構：Platform、Cognition、Harness、Memory、Task Engine、Autonomy、Extensibility。
- Harness 已有 lifecycle state machine、scope-aware permission、schema validation、post-validator、rollback、JIT retrieval、legitimacy guard。
- Memory 已有 semantic / episodic / procedural / relational、governance、contradiction detection、decay、MemoryPulse、MemoryFacade。
- TUI / Discord / CLI 已經不是薄殼，而是逐步承載 control surface 與 execution envelope。
- 架構測試已開始禁止 core 反向依賴 platform，代表專案開始自覺地管理邊界。

同時也看到一些警訊：

- 文件的世界觀更新速度快於、也有時落後於實作。例如舊文件曾描述不存在的 `DecisionPipeline`，雖然後續已有 audit 修正，但這代表需要制度化防 drift。
- `LoomSession` 仍是巨大中樞，且已在檔案開頭明示「core session lazy import platform cli tools」是 transitional coupling。
- 安全層目前有 CommandScanner 與 self-termination guard，但真正 OS/container sandbox 仍是未實作的 Issue #29。
- 記憶治理已經很前沿，但 relational memory、dreaming、skill evolution 若缺少評估基準，容易從「學習」滑向「自我污染」。

## Loom 的核心優勢

### 1. Harness-first 是正確的長期賭注

多數 agent 框架把工具當函數，Loom 把工具當有生命週期的行動。這是根本差異。

`LifecycleMiddleware` + `LifecycleGateMiddleware` 的兩層設計很成熟：外層處理紀錄、回滾、memorialize，內層處理真正執行前的 control gate。這讓 Loom 能回答三個 demo 框架通常答不出的問題：

- 這次行動誰批准？
- 行動真的達成語義目的了嗎？
- 如果沒有，系統如何回滾與學習？

這是 Loom 最該繼續深化的護城河。

### 2. Memory-native 不是口號，已經有治理雛形

Loom 不只是存 embedding。它有 trust tier、admission gate、contradiction detector、decay cycle、self-portrait triples、skill genome。這些元素組合起來，方向接近「agent 的長期人格與操作直覺」。

最值得保留的設計是：記憶不是單向累積，而是會被治理、衰減、矛盾解決與反思。這避免了很多 long-term memory agent 最終變成垃圾場。

### 3. Control surface 已開始跨平台統一

CLI、TUI、Discord 都在承載同一套 confirmation、envelope、grant、verdict、tasklist。這是成為真正 operator-grade agent 的必要條件。

下一步不該只是「更多漂亮 UI」，而是讓所有 UI 都能投影同一個 execution graph 與 action ledger。

## 最重要的架構風險

### 1. Session 中樞過重

`loom/core/session.py` 目前是所有 subsystem 的交會點：model routing、memory wiring、tool registration、middleware pipeline、telemetry、prompt stack、platform callbacks、turn streaming 都集中在這裡。

這在早期很合理，但未來會讓每個新能力都被迫穿過同一個檔案。建議把 `LoomSession` 逐步收斂為 orchestration shell，將可替換的責任拆成明確 port：

| Port | 責任 |
|------|------|
| `ToolSurface` | 註冊 core tools、platform tools、plugin tools |
| `TurnRuntime` | 處理 stream_turn 狀態機與 interruption |
| `MemoryRuntime` | 組裝 facade、pulse、governor、compression |
| `ControlRuntime` | confirmation、scope grants、abort、pause/resume |
| `EventRuntime` | 對外 emit normalized events |

目標不是大重構，而是每次碰到對應功能時抽出一條邊界。

### 2. 安全邊界仍不夠硬

CommandScanner 的文件說得很準：它是 tripwire，不是 sandbox。這個誠實很重要。

未來 Loom 若要執行更自治的任務，`run_bash`、web fetch、MCP tool、subagent 都會擴大攻擊面。下一階段應把 Issue #29 提升成 P0 級底盤能力：

- workspace mount allowlist
- network egress policy
- environment secret projection policy
- per-tool filesystem capability
- container / lightweight VM backend
- artifact output channel 與 source provenance

一個簡單方向：先做 `ExecutionBackend` 介面，預設仍是 local subprocess，但允許 Docker / sandbox-exec / firejail / gVisor 之類後端實作。先有抽象，再逐步加硬。

### 3. 記憶演化缺少評估閉環

Loom 已經能反思、dream、mutate skill、寫 self triples。這很強，但也危險。

當 agent 可以修改自己的 procedural memory，系統需要一套「記憶 CI」：

- 新 skill candidate 必須有 replay case 或 synthetic eval。
- Dreaming 生成的 relational triple 需要低 trust tier、可撤銷、可追蹤來源。
- Self-portrait 應避免短期失敗造成永久人格偏移。
- 每次 promoted skill 要能回答：它在哪些任務上提升了什麼指標？

建議把 Memory Governance 從「寫入治理」推進到「演化治理」。

### 4. 文件與實作 drift 會拖慢後期速度

Loom 的文件很多，這是優點。但多文件專案最怕文件成為平行宇宙。

建議新增一個文件一致性測試層：

- AST 掃描 doc 中的 `class X` / `function X` / `loom.foo.bar` 引用是否存在。
- doc 中標註「已實作」的功能必須有對應測試或 symbol。
- `doc/SUMMARY.md` 的版本號與 `pyproject.toml` / README version history 定期檢查。
- 每次 release 自動產生「文件 drift 報告」。

這會讓文件繼續成為資產，而不是維護負債。

## 未來三階段 Roadmap

### Phase 1：收斂底盤，讓 Loom 更可信

建議週期：1-2 個月。

| 優先級 | 主題 | 目標 |
|--------|------|------|
| P0 | OS-level sandbox | `run_bash` 從 cwd sandbox 升級為真正隔離後端 |
| P0 | Session 分層 | 從 `LoomSession` 抽出 ToolSurface / EventRuntime / MemoryRuntime |
| P1 | Doc drift CI | 文件引用 symbol 檢查、README/doc/version 同步檢查 |
| P1 | Security policy manifest | 每個 tool 宣告 fs/network/env/process 能力 |
| P1 | Eval harness | 為 memory、skill evolution、judge、autonomy 建立 replay set |
| P2 | Event schema freeze | 將 `loom/core/events.py` 定義成跨平台穩定契約 |

這階段的成功標準不是功能變多，而是「agent 做過的事更可驗證」。

### Phase 2：把事件、記憶、任務統一成時間線

建議週期：2-4 個月。

目前 Loom 有 ActionRecord、ExecutionEnvelope、EpisodicMemory、SessionLog、TaskList、Telemetry。它們都在描述同一件事：agent 在時間中如何行動。

建議建立一個統一的 `AgentLedger` 概念：

```text
Turn
  ├─ Thought / model event
  ├─ Tool action lifecycle
  ├─ Permission decision
  ├─ Memory read/write
  ├─ TaskList mutation
  ├─ Judge verdict
  └─ User-visible artifact
```

所有平台 UI 只讀 Ledger 投影：

- CLI 顯示線性事件流
- TUI 顯示 dashboard / envelope / timeline
- Discord 顯示 condensed cards
- API 回傳 JSON trace
- Memory compression 從 Ledger 提取 facts
- Replay/eval 用 Ledger 重跑任務

這會讓 Loom 從「多個 subsystem 並存」變成「同一條語義時間線的多種投影」。

### Phase 3：走向多代理社會，而不是只做 subagent

建議週期：4-9 個月。

目前 subagent 是 bounded、ephemeral、parent-controlled。這是安全起點，但未來可以發展成 agent society：

| 能力 | 描述 |
|------|------|
| Agent roles | researcher、operator、critic、security reviewer、maintainer |
| Shared blackboard | 子代理透過 ledger / scratchpad 發佈中間發現 |
| Debate protocol | 高風險決策前由 critic agent 反駁 |
| Specialist memory | 不同 agent 擁有不同 procedural memory 與 failure profile |
| Consensus gate | destructive / autonomous actions 要多代理簽核 |

關鍵不是讓代理數量變多，而是讓「分工、衝突、證據、共識」都進入 harness lifecycle。

## 大膽但值得押注的能力

### 1. Memory Diff 與 Belief Git

把 semantic / relational memory 當成可 diff、可 branch、可 rollback 的 belief graph。

未來使用者可以問：

- 你從什麼時候開始相信這件事？
- 哪次工具結果改變了你的判斷？
- 如果撤銷昨天的 dreaming，今天的建議會變嗎？
- 這個 skill 版本為何被 promoted？

這會讓 Loom 的記憶不只是 recall，而是可審判的 epistemic ledger。

### 2. Capability Firewall

把 TrustLevel 從工具級別升級成 capability policy：

```toml
[policy.network]
default = "deny"
allow_domains = ["api.github.com"]

[policy.filesystem]
write_roots = ["doc/", "tests/fixtures/"]
deny_globs = [".env", "**/secrets/**"]

[policy.env]
expose = ["PATH"]
redact = ["*_API_KEY", "*TOKEN*"]
```

這會比 SAFE / GUARDED / CRITICAL 更適合自治任務，也能支援企業部署。

### 3. Tool Contract Tests Generated From Registry

每個 `ToolDefinition` 都能自動生成 contract tests：

- schema accepts valid examples
- rejects hallucinated args
- scope_resolver 不逃逸 workspace
- post_validator 能抓到 hollow success
- rollback_fn 能恢復狀態
- output masking 不洩漏 secret

這會把 harness-first 的理念落成測試工廠。

### 4. Operator Replay

讓使用者可以重播某次 session：

- 用同一份 session log、同一份 tool result、不同模型重跑 reasoning。
- 對比「模型變了，工具決策是否變了」。
- 用 replay 來測試新 skill 是否改善舊失敗。

這會讓 Loom 成為可回歸測試的 agent runtime，而不是一次性聊天程式。

### 5. Continuous Context

目前多數 agent 仍是 turn-based。未來模型會越來越接近 continuous stream：語音、視覺、browser、filesystem watcher、notification 都是事件流。

Loom 可以提前準備：

- 把 `stream_turn()` 抽象為 event loop consumer。
- 將 user message、tool observation、timer、external event 都視為 `InputEvent`。
- 讓 abort / pause / confirmation 變成 runtime control events。

這會讓 Loom 更容易接住未來的 realtime model。

## 具體建議清單

### 立即可做

1. 新增 `doc/51-Agent-Ledger-設計.md`，正式定義 Turn / Action / Memory / Task / Artifact 的統一事件模型。
2. 新增 doc drift test，先檢查文件中的 `loom.xxx` 路徑是否存在。
3. 將 `loom.core.session` 對 `loom.platform.cli.tools` 的 transitional coupling 建 issue 並拆出第一個 `core.tools` 模組。
4. 為 `_resolve_workspace_path`、scope resolvers、`run_bash` 補 security regression tests。
5. 為 skill evolution 建立最小 replay eval：失敗 trace -> candidate -> shadow -> promotion metric。

### 中期應做

1. 實作 `ExecutionBackend`，讓 `run_bash` 支援 local / sandbox backend。
2. 將 MemoryPulse hook 擴展成可配置的 signal budget，避免 proactive memory 變噪音。
3. 把 post-validator 結果寫入統一 ledger，讓 semantic failure 可被 replay 與統計。
4. 讓 TUI / Discord / CLI 對同一份 event contract 做 rendering，而不是各自理解 runtime 細節。
5. 為 autonomy 建立 dry-run simulator，所有 cron/event trigger 可先演練再啟用。

### 長期押注

1. Belief Git：記憶分支、diff、rollback、merge。
2. Agent society：共享 blackboard、多代理 review、consensus gate。
3. Multimodal tool lifecycle：圖片、影片、browser state、GUI action 都進入 harness。
4. Policy-as-code：capability firewall、enterprise approval workflow。
5. Runtime replay marketplace：每個 bug 都能變成 agent eval case。

## 對 Loom 定位的建議

Loom 最適合的定位不是「更好用的聊天 agent」，而是：

> A local-first, auditable agent runtime for long-lived, tool-using, self-improving agents.

中文可以寫成：

> 一個本地優先、可審計、可演化的長期 agent runtime。

這個定位有三個好處：

- 避免跟 LangChain / AutoGen / CrewAI 在「誰整合更多工具」上內耗。
- 把 Loom 的差異化放在 harness、memory governance、operator control。
- 能自然延伸到個人長期代理、企業內部代理、研究型代理、自治維運代理。

## 最終評語

Loom 的方向非常有生命力。它最打動人的地方，是沒有把 agent 當成一次性 prompt，而是把它當成一個會留下痕跡、會犯錯、會修正、會累積直覺的長期行動者。

下一階段要小心的不是「功能不夠多」，而是「太多好點子同時長大，卻沒有統一的 runtime contract」。只要把事件帳本、安全沙盒、記憶演化評估、文件 drift CI 這四件事打牢，Loom 就會從一個很有想法的 agent framework，進化成真正可信的 agent operating substrate。
